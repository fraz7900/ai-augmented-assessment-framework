# SOC 2 Expert

Grounding knowledge for working with SOC 2 (the AICPA Trust Services Criteria) in this project. Load this skill before editing `framework_mapping/soc2_tsc.yaml`, SOC 2-specific mapping logic, or any content that claims to represent SOC 2 structure.

## The defining constraint: freely downloadable is not the same as freely licensed to reproduce

**The AICPA's 2017 Trust Services Criteria (TSC) — the real control criteria a SOC 2 report is assessed against — is copyrighted, all-rights-reserved content**, confirmed two ways before any code was written: the AICPA's own copyright policy page states permission is required to reproduce or redistribute AICPA content, and the source PDF's own final page states "© 2020 Association of International Certified Professional Accountants. All rights reserved." This is true **even though the document itself is freely downloadable at no cost** (unlike ISO/IEC 27001:2022's paywall, ADR-0024) — "free to download" and "licensed for reproduction" are different questions, and only the second one determines this project's transcription scope. Confusing the two would be the mistake here, not fabricating content from memory.

Given that, this framework got the same **titles-only-equivalent treatment as ISO 27001** (ADR-0024), adapted to how the TSC is actually structured: each criterion's own short **STATEMENT** is transcribed verbatim (e.g. `"The entity implements controls to prevent or detect and act upon the introduction of unauthorized or malicious software to meet the entity's objectives."` for CC6.8) — the shortest real, meaningfully-identifying unit for a TSC criterion, the same role a title plays for an ISO 27001 control. **The much longer "points of focus" that follow every criterion in the real document (illustrative bullet lists, often 5–15 per criterion) are deliberately NOT transcribed** — that is the substantial, clearly copyrightable elaboration, analogous to ISO 27001's full "shall" requirement text.

**Every consumer of `framework_mapping/soc2_tsc.yaml` must respect this**: `Practice.text` is the criterion statement only, never the fuller points-of-focus detail the real standard also contains. `FrameworkDefinition.scoring_note` states this explicitly — read it before writing any code or documentation that presents SOC 2 data to a user, and never imply a criterion statement is the complete guidance a real SOC 2 audit would apply.

## Model structure

- **5 Trust Services Categories, 61 criteria total**: Security (33, via the "Common Criteria" CC1–CC9 series — Control Environment, Communication and Information, Risk Assessment, Monitoring Activities, Control Activities, Logical and Physical Access Controls, System Operations, Change Management, Risk Mitigation), Availability (3, A1.1–A1.3), Confidentiality (2, C1.1–C1.2), Processing Integrity (5, PI1.1–PI1.5), Privacy (18, across P1–P8 sub-sections). Modeled as one `Domain` per category (short_code `"CC"`/`"A"`/`"C"`/`"PI"`/`"P"`, matching each category's real criteria-ID prefix), one `Objective` per CC-series or per Privacy sub-section (Availability/Confidentiality/Processing Integrity have no further subdivision, one flat `Objective` each), and one `Practice` per criterion.
- `scoring_model` is `"coverage"` (mirrors NIST CSF 2.0/NERC CIP/ISO 27001/CIS Controls) — SOC 2 has no native maturity-level concept. `Practice.mil` is always `None`.
- `Practice.applicability` records whether a criterion is one of the **33 Common Criteria** (`"Required in every SOC 2 report (Common Criteria / Security baseline)"` — required regardless of engagement scope) or one of the **28 additional category-specific criteria** (`"Required only when {category} is included in the engagement's scope"`) — a real distinction verified from the source document's own applicability table (TSP Section 100, para. .07), the same reuse of the applicability concept ADR-0021 (NERC CIP impact tiers) and ADR-0025 (CIS Controls Implementation Groups) already established, not a new schema concept.
- All 5 categories are fully populated (`practices_populated: True` throughout) with real criterion IDs and statements — this is **not** a partial-transcription state like NERC CIP's original stub start (ADR-0021). Every criterion's ID and statement is real and complete; what's absent is the points-of-focus elaboration for all of them, disclosed once at the framework level via `scoring_note`, not per-practice.

## Verification method (confirmed, don't re-litigate)

- The AICPA's official TSC download page requires free registration; a direct third-party-hosted mirror of the same real PDF was used instead and verified via `pypdf` — confirming genuine "TSP Section 100" branding and the exact copyright notice quoted above — before extracting anything from it, the same "verify the actual content, don't trust the URL" discipline ADR-0025 applied to CIS Controls' third-party-hosted PDF.
- The mirror obtained is the 2017 TSC as amended March 2020; AICPA's own public statements describe the more recent "2022 revised points of focus" edition as changing **only** the points of focus, not the criteria statements themselves — since this transcription excludes points of focus entirely, the criteria text here is accurate regardless of which points-of-focus edition it was sourced alongside.
- The 61-criterion total and the 5-category/9-CC-series structure were confirmed directly from the source PDF's own text (grep-verified criteria-ID patterns and section headers — "CONTROL ENVIRONMENT," "LOGICAL AND PHYSICAL ACCESS CONTROLS," etc.), not assumed from general SOC 2 familiarity.

## Rules for this project

1. **Never present a criterion statement as if it were the full SOC 2 guidance.** Any UI, report, or documentation surface showing SOC 2 data must make clear that only the criterion statement is verified/available, not the points-of-focus elaboration a real audit would also apply.
2. **Never hardcode SOC 2 structure in Python.** It belongs in `framework_mapping/soc2_tsc.yaml`, generated by `backend/scripts/generate_soc2_yaml.py`, per ADR-0002.
3. **Do not attempt to fill in points-of-focus text from memory, ever**, even if you are confident you know it. This applies the exact same discipline ADR-0024 established for ISO 27001's full requirement text.
4. **Cross-framework equivalence involving SOC 2 is methodologically weaker than the C2M2/NIST/CIS Controls pairings** (see `framework_mapping/cross_framework_equivalence.yaml`'s own header and ADR-0026) — these are statement-level topical judgments, not the full-text-vs-full-text semantic comparison those pairings use. Disclose this explicitly wherever SOC 2 equivalence entries are surfaced; treat it with the same caution as the ISO 27001 pairing, not the same confidence as the CIS Controls one.
5. **SOC 2's Common Criteria explicitly cover physical access control (CC6.4/CC6.5)** — a genuinely better fit for NERC CIP's CIP-006 (Physical Security) than CIS Controls, which has almost no physical-security coverage (ADR-0025). Don't assume SOC 2 is uniformly weaker than CIS Controls just because its overall NERC CIP hit rate is lower; the specific areas of relative strength and weakness differ, and should be checked per-standard, not assumed from the aggregate count.
6. **Several genuine gaps exist at the criterion-statement level that would NOT exist if points of focus were in scope**: personnel background-check/screening, patch management, password composition/rotation, and general audit-log retention are all mentioned only in points of focus, never in a bare criterion statement — don't force an equivalence match for these concepts against a SOC 2 criterion whose statement doesn't actually cover them, even if a plausible-sounding criterion exists nearby.

## Example usage

Extending SOC 2 equivalence to a new framework, or reviewing NERC CIP↔SOC 2 more thoroughly: load this skill first to confirm which criteria exist and their real Common/Additional applicability before making any claim about "what SOC 2 requires" — the honest answer for any mechanism only covered in points of focus (e.g. "does SOC 2 require a specific password rotation interval?") is "not part of the verified criterion statement in this project's data," not a guess filled in from general SOC 2 familiarity.
