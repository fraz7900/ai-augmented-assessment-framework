# PCI DSS Expert

Grounding knowledge for working with the Payment Card Industry Data Security Standard (PCI DSS) v4.0.1 in this project. Load this skill before editing `framework_mapping/pci_dss_v4.yaml`, PCI-specific mapping logic, or any content that claims to represent PCI DSS structure.

## The defining constraints: copyright (like ISO 27001/SOC 2) AND a genuinely deeper structure (unlike anything else)

**PCI DSS v4.0.1 is copyrighted, all-rights-reserved content**, confirmed directly: the source PDF's own page 1 states "©2006 - 2024 PCI Security Standards Council, LLC. All Rights Reserved," and no public PCI SSC reproduction license was found (unlike CIS Controls' Creative Commons license, ADR-0025). This is true even though the document is freely downloadable at no cost — the same "free to download" vs. "licensed for reproduction" distinction ADR-0026 established for SOC 2. Given that, this framework got the same statement-only treatment as ISO 27001/SOC 2, not CIS Controls' full-transcription treatment.

**PCI DSS also has a real structural difference from every other framework in this project: it is THREE levels deep**, not two. A top-level Requirement (e.g. "Requirement 1: Install and Maintain Network Security Controls") breaks into Sections (e.g. "1.1 Processes and mechanisms for installing and maintaining network security controls are defined and understood."), which in turn break into individual numbered "Defined Approach Requirements" (e.g. 1.1.1, 1.1.2, 1.2.1-1.2.8...) — roughly 205 of these leaf items exist across all 12 Requirements, each with its own Testing Procedures, Purpose, Good Practice, Examples, Definitions, and Customized Approach Objective elaboration.

Given this exceptional depth (no other framework here has three numbered levels, and 205 leaf items would be a materially larger single-pass transcription than any framework transcribed so far), **this project deliberately transcribes at the SECTION (N.N) level, not the finer Defined Approach Requirement (N.N.N) level** — a real, disclosed scope decision (which granularity counts as this project's atomic "Practice") that is distinct from, and in addition to, the copyright-driven "statement not full text" decision. See ADR-0027 for the full disclosure, including the ~205 leaf-level items being named as real, unstarted-deeper-granularity future work.

**Every consumer of `framework_mapping/pci_dss_v4.yaml` must respect this**: `Practice.text` is the Section-level statement only (e.g. `"9.2 Physical access controls manage entry into facilities and systems containing cardholder data."`), never the finer leaf-level requirement text or its Testing Procedures/Purpose/Guidance. `FrameworkDefinition.scoring_note` states this explicitly.

## Model structure

- **12 Requirements, 63 Sections total** (5+3+7+2+4+5+3+6+5+7+6+10 across Requirements 1-12: Install and Maintain Network Security Controls; Apply Secure Configurations; Protect Stored Account Data; Protect Cardholder Data with Strong Cryptography During Transmission; Protect Against Malicious Software; Develop and Maintain Secure Systems and Software; Restrict Access by Business Need to Know; Identify Users and Authenticate Access; Restrict Physical Access; Log and Monitor All Access; Test Security Regularly; Support Information Security with Organizational Policies). Modeled as one `Domain` per Requirement (short_code `"REQ-01"`-`"REQ-12"`, zero-padded), one `Objective` per Requirement (no further subdivision — PCI DSS's real intermediate level, the Section, is itself the Practice here, the same "single Objective per Domain" pattern ISO 27001 uses when there's no further real subdivision to model), and one `Practice` per Section.
- `scoring_model` is `"coverage"` (mirrors every non-C2M2 framework here) — PCI DSS has no native maturity-level concept. `Practice.mil` is always `None`.
- `Practice.applicability` is always empty — no per-Section applicability-scope concept was verified in the source (PCI DSS does have SAQ-type engagement scoping, but it doesn't map cleanly to a per-Section field the way NERC CIP's impact tiers, CIS Controls' Implementation Groups, or SOC 2's Common/Additional-category distinction do — don't fabricate one).
- All 12 Requirements are fully populated (`practices_populated: True` throughout) at the Section level — this is **not** a partial-transcription state like NERC CIP's original stub start (ADR-0021). Every Section's ID and statement is real and complete; what's absent is the finer leaf-level Defined Approach Requirement detail for all of them, disclosed once at the framework level via `scoring_note`.

## Verification method (confirmed, don't re-litigate)

- Fetched the real "Payment Card Industry Data Security Standard: Requirements and Testing Procedures, v4.0.1" PDF (June 2024 — the current active version; v4.0 was retired 31 December 2024) from a third-party institutional mirror, verified via `pypdf` to be genuine PCI SSC content (real title page, the exact copyright notice quoted above, 397 real pages) before extracting anything from it, the same "verify the actual content, don't trust the URL" discipline ADR-0025/ADR-0026 established.
- The 63-Section total was confirmed two independent ways: an automated grep-based count of the document's own N.N-pattern numbering, and a manual read of each of the 12 Requirements' own real "Sections" summary block (a short list at the very start of each Requirement, which conveniently already contains the complete, real Section statement text — no need to wade through leaf-level Testing Procedures detail to get this).

## Rules for this project

1. **Never present a Section statement as if it were the full PCI DSS requirement.** Any UI, report, or documentation surface showing PCI DSS data must make clear that only the Section-level statement is verified/available, not the finer Defined Approach Requirement text or its Testing Procedures/Purpose/Guidance.
2. **Never hardcode PCI DSS structure in Python.** It belongs in `framework_mapping/pci_dss_v4.yaml`, generated by `backend/scripts/generate_pci_dss_yaml.py`, per ADR-0002.
3. **Do not attempt to fill in leaf-level Defined Approach Requirement text from memory, ever**, even if you are confident you know it — the same discipline ADR-0024/ADR-0026 established for ISO 27001/SOC 2's excluded text.
4. **Cross-framework equivalence involving PCI DSS is methodologically weaker than the C2M2/NIST/CIS Controls pairings** (see `framework_mapping/cross_framework_equivalence.yaml`'s own header and ADR-0027) — these are Section-level statement judgments, not the full-text-vs-full-text semantic comparison those pairings use. Treat it with the same caution as the ISO 27001/SOC 2 pairings.
5. **Watch for "account" vocabulary collisions during any future equivalence review.** PCI DSS's "account data" almost always means payment card account data (PAN, cardholder data), while most other frameworks' "account" language means user/system accounts — the embedding will surface superficially plausible matches across this distinction that a careless reviewer could accept incorrectly. Always check what "account" means in context before accepting a candidate.
6. **PCI DSS has no dedicated business-continuity/disaster-recovery Section**, unlike SOC 2's A1.1-A1.3 or CIS Controls' Control 11 — a real, confirmed gap (PCI DSS's scope is protecting cardholder data via security controls, not business continuity). Don't force a match for a NERC CIP recovery-plan part against an unrelated PCI Section just because a superficial keyword overlaps.
7. **When adding a new framework in the future, check for practice-ID collisions with existing frameworks before assuming the loader will handle it correctly.** This project found and fixed a real bug during PCI DSS's addition: `services/framework_loader.py`'s practice-text index used to be keyed by bare `practice_id` alone, so CIS Controls' Safeguard "5.1" and PCI DSS's Section "5.1" (an identical bare ID string) silently collided — loading PCI DSS overwrote CIS Controls' entry, corrupting an existing equivalence entry's framework attribution. Fixed by keying the index by `(framework_name, practice_id)`. Any future framework whose ID scheme could plausibly collide with an existing one (short numeric-dotted forms are the highest risk) should be spot-checked against this exact failure mode after loading.

## Example usage

Extending PCI DSS equivalence to a new framework, or reviewing NERC CIP↔PCI DSS more thoroughly: load this skill first to confirm which Sections exist and that the finer leaf-level Defined Approach Requirement detail is out of scope before making any claim about "what PCI DSS requires" — the honest answer for any specific mechanism (e.g. "does PCI DSS require password rotation on a specific cadence?") is "not part of the verified Section-level statement in this project's data," not a guess filled in from general PCI DSS familiarity.
