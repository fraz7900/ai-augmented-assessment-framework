# api/

FastAPI routers only. Each router should be a thin HTTP adapter: parse the request, call a service in `services/`, return the response. No business logic, no direct database or vector store access from this layer — that dependency inversion is what keeps the API layer swappable (e.g., adding a CLI or batch interface later without duplicating logic) and testable without spinning up HTTP.

Modules: `health.py`, `ingestion.py` (Sprint 1), `assessments.py` (Sprint 2, extended with `/score` in Sprint 3, `/dashboard` in Sprint 6, `/report/pdf` and `/report/xlsx` in Sprint 7, `/chat` in Sprint 8), `frameworks.py` (Sprint 3, browse a loaded framework schema), `dependencies.py` (dependency-injection wiring, not a router — see its own docstring), `error_handlers.py` (Sprint 9, not a router — centralizes exception-to-HTTP-status mapping so endpoints don't each repeat their own try/except, see its own docstring and ADR-0015). Mapping proposals (Sprint 5) and reports (Sprint 7) were originally planned as separate `mappings.py`/`reports.py` router files; both ended up staying inside `assessments.py` instead, since they operate on the same `/assessments/{id}/...` resource — noted here because that plan changed, not silently updated.
