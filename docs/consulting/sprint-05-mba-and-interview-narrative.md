# Sprint 5 — MBA and Consulting Interview Narrative

Continues the convention from Sprints 0-4: claims here are tied to what was actually built and found this sprint, listed in `docs/consulting/sprint-05-deliverables.md`'s Change Log.

## MBA Applications

**Rigorous feasibility analysis, not optimistic planning, drove the AI architecture decision.** The project charter and prior sprints' documentation both described "local LLM reasoning" as a planned capability. Rather than either building toward that assumption or quietly abandoning it, Sprint 5 checked it directly: fetched the actual Ollama install script, read what it does (creates a system user, a systemd service, requires `sudo` throughout), and checked the sudo-free alternative's actual package size (1.4 GB) against every dependency this project has installed before (previous largest: under 20 MB). The decision to ship retrieval-only this sprint is the output of that specific comparison, not a default fallback — the kind of "verify the assumption before committing resources to it" discipline that separates a plan from a guess.

**A predicted failure mode was tested for, and found, rather than assumed away.** ADR-0011's threshold calibration explicitly stated the gap between correct and incorrect matches was "real but not large" before the live demo ran. The demo then produced exactly that: a false positive (`ASSET-1a` at 0.71 confidence, wrong domain, matched on generic vocabulary) sitting close to real correct matches (0.77-0.85). Reporting this as confirmation of a documented risk, rather than either hiding it or being surprised by it, is the difference between a risk register that's decorative and one that's actually predictive.

**A fourth consecutive sprint where the testing discipline caught something a lighter approach would have missed.** The status-history ordering bug was intermittent and non-reproducible in isolation — exactly the kind of bug that survives code review, survives a single test run, and only shows up under real, repeated integration testing. Root-causing it (traced to implicit reliance on wall-clock timestamps) and fixing it with a verified, stable solution (three consecutive clean full-suite runs, not just "the diff looks right") before moving on is the same investment-in-process logic that makes a rigorous QA function worth its cost.

## Consulting Interviews

**Consulting competency demonstrated:** making a build-vs-defer infrastructure call the same way a strategy team evaluates a vendor or platform decision — cost the actual alternative (package size, operational overhead, a dependency this environment cannot self-service without escalation), compare it against what's achievable within the current budget, and document the tradeoff so it can be revisited on its own terms later rather than re-litigated from scratch. ADR-0011's "Alternatives considered" section states explicitly this was a scale judgment, not a principled rejection of the technology.

**Business problem solved:** the platform's core value proposition — reducing the manual labor of matching evidence to hundreds of compliance practices — needed to ship this sprint, but the originally-envisioned generative approach carried a cost this environment could not absorb without external help. Rather than blocking the sprint on an escalation or silently downgrading the feature's ambition without saying so, the retrieval-only design delivers real, measurable acceleration now (a human reviews and decides, instead of searching from scratch) while keeping the more expensive generative capability as a well-scoped, sequenced future increment.

**Measurable value created:** a working mapping engine verified against real data (multiple genuine high-confidence matches, one genuine and correctly-caught false positive); a complete human-in-the-loop review workflow with enforced state transitions; 27 new tests; one production bug found and fixed with verified stability; and a documented, revisitable architecture decision that turns a "we don't have Ollama yet" gap into a named, costed, sequenced backlog item rather than an open question.

**How this would be presented to a client:** as a working v1 of the AI-matching capability shipped on schedule, with an explicit, cost-justified statement of what it does and doesn't yet do — the same way a consulting team would present a pilot automation tool that handles the bulk of a manual process correctly while flagging the residual cases that still need a human, rather than either overselling full automation or waiting to ship until the harder, more expensive version is ready.

## STAR story draft

**Situation:** The project's charter had described "local LLM reasoning" as part of the platform's AI capability since Sprint 0, and Sprint 5 was the first sprint whose scope — proposing evidence-to-practice mappings — appeared to require it.

**Task:** Deliver real, working, human-reviewed AI-proposed evidence mapping this sprint, without either committing to an infrastructure dependency whose actual cost hadn't been checked, or fabricating a "confidence" or "match rationale" the platform couldn't actually back up.

**Action:** Investigated Ollama's actual install requirements directly — found a sudo-only install path and, on the sudo-free alternative, a 1.4 GB package size disproportionate to everything else this project depends on. Rather than treating that as a blocker, redesigned the sprint around what a purely retrieval-based approach could deliver honestly: embed each uncovered practice, search only the documents already tied to the assessment (a deliberate scoping decision against cross-assessment data leakage), and propose a match only above an empirically calibrated similarity threshold — with every proposal requiring an explicit human accept/edit/reject decision before it could affect a score.

**Result:** A fully real mapping engine shipped this sprint, verified live: genuine high-confidence correct matches (up to 0.85) alongside a genuine, disclosed false positive (0.71) that the human-review step correctly caught — confirming the calibration work's own predicted precision ceiling rather than being surprised by it. 27 new tests, one real production bug found and fixed along the way, and a documented, revisitable decision (ADR-0011) that keeps the more expensive generative capability as a specific, costed future increment instead of an open-ended aspiration.
