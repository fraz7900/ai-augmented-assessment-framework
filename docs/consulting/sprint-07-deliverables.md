# Sprint 7 — Consulting Deliverables

**Sprint:** 7 — PDF/XLSX Report Generation
**Period:** 2026-07-16
**Prepared as:** end-of-sprint client-style reporting, per the project's consulting-engagement operating model

---

## Executive Summary

Sprint 7 delivered the second half of the Executive Reporting epic: the Sprint 6 dashboard
(`DashboardReport`) can now be downloaded as a board-ready PDF or a filterable XLSX
(`GET /assessments/{id}/report/pdf`, `.../report/xlsx`), closing US-6.2. Four design questions had
to be resolved before writing any export code, all documented in ADR-0013: whether export
recomputes anything (no — it is a pure rendering step over the already-verified `DashboardReport`
Sprint 6 already produced and tested); whether generated files are persisted server-side (no —
generated in memory per request, matching the "computed fresh, no caching" posture `/score` and
`/dashboard` already use); whether PDF and XLSX should be the same content in two formats (no —
they are deliberately different views, a fixed board narrative versus a filterable data appendix,
matching two genuinely different consumption patterns implied by US-6.2 itself); and how to handle
the two frameworks' transcribed source text safely in a PDF library whose core fonts only reliably
support Latin-1 (explicit character translation with a safe fallback, not a crash or silent
mangling).

Both endpoints were verified live against a running server, not just against unit tests: the PDF's
extracted text (via `pypdf`, already a project dependency) contains the real assessment name, the
real gap list, and the real resolution ranking; the XLSX opens with `openpyxl` into the four
expected sheets (Situation, Domain Scores, Gaps, Resolution) with real values, including the correct
count of flattened gap rows (71 = 36 `ASSET` + 35 `ACCESS`) and the correct resolution ranking
(fewest missing first). A mid-implementation bug — fpdf2 raising `FPDFException: Not enough
horizontal space to render a single character` when consecutive `multi_cell` calls left the cursor
off the left margin — was caught by the test suite before the live demo, not by inspection, and
fixed by pinning `new_x`/`new_y` explicitly on every text write rather than depending on the
library's own default chaining behavior.

## Client Update

**What was delivered:** `services/export_service.py` (`build_pdf_report`, `build_xlsx_report`, the
`_pdf_safe` Latin-1 safety helper); `AssessmentService.generate_dashboard_pdf` /
`generate_dashboard_xlsx`; `GET /assessments/{id}/report/pdf` and `.../report/xlsx` in
`api/assessments.py`, each with a correct `Content-Type` and a `Content-Disposition: attachment`
filename derived from the assessment's own name; `fpdf2` promoted from a `dev` to a main dependency
(it is now production code) and `openpyxl` added as a new main dependency; 9 new tests (130 → 139,
all passing), 7 unit (fakes-based, reading generated bytes back with `pypdf`/`openpyxl`) and 2 live
integration tests against the real FastAPI app.

**What was intentionally not delivered this sprint:** any server-side persistence or archival of
generated reports (deliberate — see ADR-0013's no-persistence decision and new risk R-21); charts or
visual comparisons of domain scores across frameworks (deliberate — ADR-0012 already established
C2M2's ordinal MIL and NIST CSF's continuous coverage are not comparable on one axis, and a chart
would visually imply otherwise); a Unicode TTF font bundle for fpdf2 (deferred — explicit character
translation solved the actual observed problem, a handful of em/en dashes in transcribed framework
text, without a new bundled-font dependency); any frontend UI (the export endpoints are API-only,
consistent with this project's backend-first sequencing every sprint so far).

**Decision made and disclosed, not escalated:** the choice not to persist or version exported
reports server-side, even though it means a downloaded PDF/XLSX is a point-in-time snapshot with no
platform-side record of which snapshot is in circulation once evidence is later reviewed, added, or
rejected. This is tracked honestly as new risk R-21 rather than left unstated — the mitigating
factor is that both formats print a `Generated: <timestamp>` value directly in the file, which
survives the export leaving the API even without server-side tracking.

## Architecture Decision Record — Summary

| ADR | Decision | One-line business reason |
|---|---|---|
| 0013 | Export renders the existing `DashboardReport` with no recomputation and no server-side persistence; PDF and XLSX are deliberately different layouts, not the same content twice | Sprint 6 already produced a real, verified report; recomputing it in a second place would be a maintenance/drift risk, and a board narrative and a compliance lead's working spreadsheet are genuinely different artifacts, not one report in two file extensions |

## Business Value Assessment

- **Sprint 6's investment in a real, typed `DashboardReport` paid off exactly as ADR-0012 predicted.**
  Its Consequences section named this sprint's likely shape — "a rendering problem on top of
  already-correct data, not a second computation layer to build and re-verify" — and that is
  precisely what happened: `export_service.py` contains zero scoring or gap-analysis logic, only
  presentation.
- **The platform now closes the loop US-6.2 named from the start:** a CISO (Marcus) can bring
  findings to a board with no API access, and a compliance lead (Priya) gets a working spreadsheet
  to filter and sort rather than a PDF she'd have to manually retype into Excel. Building two
  genuinely different artifacts, rather than the same layout rendered twice, is what makes both
  personas actually served, not just one persona served twice.
- **A real implementation bug was caught by the test suite before the live demo, which is itself the
  argument for writing the tests before declaring the feature done.** The `FPDFException` would have
  surfaced as a 500 error on any real report containing more than a couple of lines of text — every
  demo assessment used in this sprint's own verification would have triggered it. Finding and fixing
  it during test-writing, rather than during the live walkthrough, is the direct payoff of this
  project's standing practice of running the full suite before calling a sprint complete.

## Risk Assessment

Full register in `docs/product/risk_register.md`; summarized here for Sprint 7 changes:

| Risk | Sprint 7 status |
|---|---|
| R-20, no caching on `build_dashboard`/`compute_scores`, recomputed on every call | Unchanged, extended by design — export reuses `build_dashboard` per request rather than introducing a report cache, consistent with this project's practice of not optimizing for a problem that hasn't been measured |
| R-21 (new), a downloaded PDF/XLSX is a point-in-time snapshot with no server-side version record | Open, disclosed — direct, deliberate consequence of ADR-0013's no-persistence decision; both formats carry a `Generated:` timestamp as a self-contained staleness signal |

## ROI Estimate

- **Investment this sprint:** one new service module (`services/export_service.py`), two new
  `AssessmentService` methods, two new API endpoints, two new main dependencies (`fpdf2`, `openpyxl`
  — both pure Python, no system-level binaries), 9 new tests (7 unit, 2 live integration), one ADR
  resolving four genuine design questions, and a live demo verified by reading the generated files
  back with the same libraries a real consumer would use.
- **Return:** the platform's core value proposition — faster, evidence-grounded compliance
  assessment — now has an actual leave-the-API artifact a board or auditor can hold, not just a JSON
  response only this project's own developer has ever seen.
- **Compounding return:** because export never recomputes anything, any future improvement to
  `report_service.py`'s gap analysis or scoring (e.g., transcribing the remaining 8 C2M2 domains,
  US-3.1a) automatically improves both exported formats with zero changes to `export_service.py` —
  the rendering layer and the computation layer stay fully decoupled.
