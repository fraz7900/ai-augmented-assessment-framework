# Decision Log

Business- and product-facing decisions, in chronological order. Technical decisions with full architectural rationale live as ADRs in `docs/adr/`; this log is the shorter, cross-referenced index a non-technical stakeholder would actually read, plus decisions that aren't architectural at all.

| # | Decision | Sprint | Why | Reference |
|---|---|---|---|---|
| D-1 | MVP scope limited to C2M2 and NIST CSF 2.0; five other frameworks explicitly deferred to roadmap | 0 | Prevent scope sprawl across seven frameworks before any one of them works end to end | `PROJECT_CHARTER.md` Section 12 |
| D-2 | No multi-tenant authentication in MVP | 0 | Single-user assumption holds for the project's actual use (solo portfolio build); avoids building access-control infrastructure with no one to test it against | `PROJECT_CHARTER.md` Section 9 |
| D-3 | Repository uses src-layout, deviating from the originally briefed flat `src/` sibling directory | 0 | Prevents accidental local-import bugs; documented explicitly rather than silently changed | ADR-0001 |
| D-4 | Framework definitions and prompts are data, not code | 0 | Makes future framework additions additive rather than engine rewrites; central to the extensibility claim in the roadmap | ADR-0002 |
| D-5 | Claude Code hooks activate progressively, tied to when their dependencies actually exist | 0 | A hook that fails on every use trains the developer to ignore it | ADR-0003 |
| D-6 | Vector store is LanceDB | 1 | No bundled ML runtime; file-based, no server process | ADR-0005 |
| D-7 | MVP embeddings use a local hashed vectorizer, not a neural model, and this is explicitly interim | 1 | Zero network dependency, minimal footprint, and — the deciding factor — mathematically correct for independent per-call ingestion, which the initially-attempted `TfidfVectorizer` was not | ADR-0006 |
| D-8 | Lint-on-save hook activated as soon as `backend/pyproject.toml` existed | 1 | Fulfilled the staged-activation plan from D-5 on schedule rather than deferring indefinitely | ADR-0003 (updated), `.claude/settings.json` |
| D-9 | Relational storage is SQLite via SQLModel, not raw SQLAlchemy with separate Pydantic schemas | 2 | Single source of truth for entity shape; avoids a duplicate-schema maintenance cost discovered to be real, not hypothetical, once attempted | ADR-0007 |
| D-10 | `practice_reference` is free text in Sprint 2, deferred to a real foreign key once framework schemas exist | 2 | Let the assessment engine's state machine and evidence-linking logic be built and tested independently of framework content readiness | `docs/product/prd.md`, Epic 2 vs. Epic 3 |
| D-11 | A finalized assessment permanently rejects new evidence links; no "un-finalize and re-add" path exists in the MVP | 2 | Matches the audit-trail requirement (Diane persona) — a "finalized" state that can be quietly reopened is not actually an audit boundary | `services/assessment_service.py`, US-2.5 |
| D-12 | Full product management documentation deferred from Sprint 0/1 to Sprint 2 kickoff | 0 (deferred), 2 (delivered) | PRD/user-story writing is more grounded once a real ingestion path and a resolved vector-store decision exist to write requirements against | `docs/product/README.md` |
| D-13 | Default embedding backend upgraded from a hashed vectorizer to a local ONNX semantic model (fastembed, `BAAI/bge-small-en-v1.5`), before Sprint 3 began | Between 2 and 3 | Directly requested resequencing following the RICE analysis in `docs/product/prioritization.md`, which scored this as the highest-leverage remaining backlog item; verified with a live retrieval test, not shipped on the strength of the RICE score alone | ADR-0008; closes R-10 in `docs/product/risk_register.md` |
| D-14 | C2M2 v2.1 encoded with only 2 of 10 domains fully transcribed (`ASSET`, `ACCESS`), rather than either fabricating the rest or blocking the sprint until all 10 were done | 3 | Verified, real data for 2 domains is more defensible than fabricated or paraphrased data for 10, given this project's standing rule against unverifiable claims; `Domain.practices_populated` makes the gap machine-readable rather than hidden | ADR-0009; opens risk R-14 and backlog item US-3.1a |
| D-15 | Source PDF (energy.gov) was parsed locally with `pypdf` after `WebFetch` failed to decode it, rather than falling back to unverified memory of C2M2's structure | 3 | The project already had a PDF-parsing dependency for its own ingestion pipeline; using it to ground the framework data in the actual source text was strictly better than proceeding on an unverifiable summary alone | ADR-0009 |
| D-16 | NIST CSF 2.0 is scored by coverage (0.0-1.0 fraction), not forced into C2M2's cumulative MIL0-3 model | 4 | NIST CSF 2.0 does not natively define maturity levels; inventing a maturity scale for it would misrepresent the actual standard, which is exactly the failure mode the `nist-csf-expert` skill was written to prevent | ADR-0010 |
| D-17 | All 6 NIST CSF 2.0 functions and all 106 subcategories were transcribed (full coverage), unlike C2M2's partial (2 of 10) coverage | 4 | The entire CSF Core fit within the single fetched source document (32 pages vs. C2M2's 96), making full transcription actually achievable within one sprint — not a relaxation of the "verified over fabricated" standard from D-14, just a case where both were simultaneously possible | ADR-0010; the generator script asserts the transcribed count matches NIST's own stated total (106) before writing the file |
| D-18 | Sprint 5's mapping engine ships retrieval-only (embed practice text, search assessment-scoped documents, propose above a calibrated threshold); Ollama-based generative extraction is deferred, not built | 5 | Actual feasibility was checked, not assumed: the sudo-requiring install script and a 1.4 GB sudo-free package size were both confirmed directly before deciding, the same "verify before deciding" discipline D-15 applied to a source document, applied here to infrastructure. A fully real, useful retrieval-only engine was achievable without it | ADR-0011; opens risk R-16, closes nothing on R-1 (downgraded, not closed, since a future generative layer would reintroduce it) |
| D-19 | Evidence-review query ordering (`status_history`, `evidence_for_assessment`) switched from implicit/timestamp-based ordering to SQLite's `rowid` | 5 | An intermittent test failure showed wall-clock timestamps are not a reliable ordering key when writes land close together; rowid is monotonically increasing per insert regardless of clock behavior — found and fixed the same sprint it was discovered, not deferred | ADR-0011; closes risk R-18 |
| D-20 | Document-scoped retrieval (`search_within_documents`), not global-corpus retrieval, was built as the mapping engine's search boundary from the start | 5 | Searching the entire vector store would let one assessment's AI-proposed evidence cite documents uploaded for a completely unrelated assessment — a cross-assessment data-leakage-shaped bug avoided structurally rather than by convention, consistent with `PROJECT_CHARTER.md` Section 7's privacy commitments | ADR-0011 |
| D-21 | Sprint 6's dashboard is entirely templated (Python f-strings from real computed numbers), not generated by a model, even though it is the platform's first true "report" surface | 6 | Nothing about the executive-reporting skill's situation/complication/resolution structure requires generated text — every sentence is directly traceable to a number computed two lines earlier. Using an LLM here would be capability theater with no functional benefit, and this project has a standing rule against unverifiable AI claims | ADR-0012 |
| D-22 | The dashboard's overall-progress headline never averages a `cumulative_mil` framework's per-domain MIL scores; it reports a count ("N of M domains at MIL1+") instead. `coverage` frameworks get a legitimate weighted-average fraction | 6 | MIL is an ordinal scale (c2m2-expert skill); averaging it into one number would misrepresent it the same way R-15 already warned against for a single domain's score, extended here to the whole-assessment headline — the single most visible number a non-technical reader would see | ADR-0012; closes R-15 for the dashboard surface specifically |
| D-23 | The resolution list's prioritization is strictly effort-based (fewest missing practices first), not a fabricated business-impact score | 6 | The platform has no real criticality or cost data in `framework_mapping/` to compute an honest impact number; effort (a real, countable quantity) is the only dimension currently supportable without inventing one | ADR-0012; opens risk R-19 |
| D-24 | PDF/XLSX export renders the existing `DashboardReport` with no new computation and no server-side persistence; PDF and XLSX are deliberately different layouts (narrative vs. flat data appendix), not the same content twice | 7 | Sprint 6 already produced a real, typed, structured report; recomputing anything here would duplicate already-settled scoring/grouping logic in a second place. No persistence matches the existing "computed fresh, no caching" posture (R-20) rather than introducing new state with no observed need. PDF (board narrative) and XLSX (filterable data) serve genuinely different consumption patterns per US-6.2's own framing | ADR-0013; closes backlog item US-6.2 |
| D-25 | Sprint 8's chat feature stays retrieval-only; the Ollama sudo blocker from ADR-0011 was re-verified rather than assumed unchanged, found to have a viable sudo-free workaround this time, and the choice between that path and retrieval-only was put to the project owner directly (`AskUserQuestion`), not decided silently | 8 | The feasibility facts changed materially since Sprint 5 (network/disk confirmed ample, a sudo-free tarball path confirmed viable), which meant "stay retrieval-only" was now a preference rather than a forced outcome — exactly the kind of consequential, hard-to-reverse choice (multi-GB download, a new persistent process, this codebase's first generative-AI capability) this project's working style reserves for explicit confirmation | ADR-0014; closes backlog item US-7.1; opens risks R-22, R-23 |
| D-26 | Sprint 9's scope (testing/refactoring/documentation) was derived from measurements taken first — a timing run, a coverage report, a manual duplication scan — rather than decided in advance; exception-to-HTTP-status handling was centralized into one new `api/error_handlers.py` module, removing ~25 duplicated try/except blocks, while a "split up the large service file" instinct was checked against actual evidence and dropped when none was found | 9 | A sprint with no fixed feature to build risks unfocused busywork; grounding it in real numbers (a 46% measured test-runtime win, a 79%→97% coverage gap on one specific file, a counted 12x/5x/3x exception-handling duplication) kept the work targeted and each change independently justified, the same "fix measured problems" discipline R-13/R-20 already established | ADR-0015; closes risk R-13 |
| D-27 | The frontend (zero React code prior to this sprint) was built as Vite + React 19 + TypeScript, react-router, TanStack Query, and types generated directly from the backend's own OpenAPI schema; Tailwind CSS was chosen over CSS Modules/a component library after asking the project owner directly (`AskUserQuestion`) rather than deciding unilaterally on a purely visual/subjective call | 10 | Build tooling had been explicitly deferred twice (Sprint 6, then again in `frontend/README.md`) with no repo precedent; each choice was matched to what this specific app actually needs (no auth, no SSR, no real-time) rather than to what's currently popular, and generating types from the live OpenAPI schema keeps the frontend's model of the API honest by construction, the same "data as code, not duplicated" principle D-4 already established for framework definitions | ADR-0016 |
| D-28 | Backend gained CORS middleware (restricted to the Vite dev-server origins, not a wildcard) so the new frontend could call it at all; `vite.config.ts` was changed to poll for file changes after a live verification pass showed Vite's default watcher silently serves stale code on this repo's OneDrive-synced `/mnt/c` WSL2 mount | 10 | Both were found by actually running the app end-to-end against the real backend during this sprint, not assumed in advance — the file-watching bug in particular produces no error at all, only wrong behavior, and was caught only because a live browser walkthrough with console-error checking was run rather than stopping at `npm run build` | ADR-0016; connects to risk R-11 |
| D-29 | Docker Compose deployment stack: one named volume (not several), Ollama defined but gated behind a Compose `profile` rather than started by default, `python:3.12-slim` (not Alpine) for the backend image, and the frontend's API base URL baked in at build time to the backend's published host port (not a Compose-internal service name) | 10 | Both the Ollama gating and "Compose orchestrates, not standalone Dockerfiles" were the project owner's explicit direction, not a unilateral call; the other choices were each traced to a concrete finding — a repo-wide grep confirmed no runtime code calls Ollama, `pyproject.toml`'s dependency list confirmed several packages don't reliably ship musl wheels, and the frontend JS running in the browser (outside the Compose network) makes a Compose service name the wrong URL by construction, not just a style preference | ADR-0017 |
| D-30 | The Docker Compose stack was built and statically verified (compose YAML parsed, every env var cross-checked against the real `Settings` class, `pip install .` replicated in an isolated venv) but explicitly **not** run end-to-end via `docker compose up`, because the authoring environment has no Docker installed and no passwordless `sudo` to install it | 10 | Consistent with this project's standing "verify before claiming" discipline (ADR-0013, ADR-0015, ADR-0016) — claiming full live verification here would have been the exact failure mode that discipline exists to prevent; the gap is named specifically rather than presented as done | ADR-0017; `deployment/README.md`'s "Known gap" section |
| D-31 | The remaining 8 C2M2 domains (285 of 356 practices) were transcribed from the source PDF, closing US-3.1a; 3 objectives with a MIL-level label silently dropped by `pypdf`'s extraction were caught by a systematic scan for a structural artifact (a stray leading space), not by trusting the raw extraction, and cross-checked against secondary sources before transcribing | 10 | Executed exactly as ADR-0009 had already documented as "a known, repeatable process" — zero changes needed to the loader/scoring/validation code, confirming ADR-0002's data-as-code separation paid off as designed. The anomaly-detection step matters because an unverified guess at even 12 of 356 practices' MIL level would have quietly reintroduced the "verified over fabricated" risk ADR-0009 was written to prevent | ADR-0018; closes risk R-14 and backlog item US-3.1a |
| D-32 | Risk R-24 (Docker Compose stack not live-verified) was closed the same sprint, after the project owner installed Docker Desktop and enabled its WSL2 integration; the live run found and fixed one real bug (`npm ci` needed `--legacy-peer-deps`) that the earlier static-only verification could not have caught | 10 | Directly confirms this project's "verify before claiming" discipline was the right call rather than excessive caution — the very first real build hit a genuine issue static review had missed, and a full persona walkthrough against the real containers (349 AI mapping candidates proposed, reflecting ADR-0018's full C2M2 transcription) then passed cleanly | ADR-0017 (updated); closes risk R-24 |
| D-33 | Cross-framework equivalence (US-5.2/FR-14) ships as computed candidates + hand-curated acceptance (79 of 106 NIST subcategories), not an automatic similarity-threshold dump — a hard constraint found in `.claude/skills/framework-mapping/SKILL.md` before writing any code, not a preference | 10 | The review itself caught a real, unanticipated false-positive pattern (C2M2's near-identical "Management Activities" boilerplate repeated across all 10 domains matching generic NIST governance subcategories equally) and several genuine matches the raw embedding ranking scored lower than false positives, which pure automation would have gotten wrong in both directions | ADR-0019; closes backlog item US-5.2/FR-14 |
