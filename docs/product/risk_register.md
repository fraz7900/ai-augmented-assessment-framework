# Risk Register

Master, living register. Consolidates `PROJECT_CHARTER.md` Section 7 (initial risks) with risks found during Sprint 1-2 execution (`docs/consulting/sprint-0*-deliverables.md`). Likelihood/impact are qualitative (Low/Medium/High) — no incident data exists yet to quantify them, and presenting false-precision numbers would contradict this project's own stated documentation standard (see `PROJECT_CHARTER.md` Section 2's business problem framing).

| ID | Risk | Category | Likelihood | Impact | Status | Mitigation |
|---|---|---|---|---|---|---|
| R-1 | AI hallucinates a compliance claim not supported by evidence | AI governance | Low (downgraded, Sprint 5) | High | **Mitigated by construction, Sprint 5** | Sprint 5's mapping engine is retrieval-only, not generative (ADR-0011): a proposed mapping's "citation" is the literal retrieved chunk, never a model-generated quote, so there is nothing to hallucinate; human-in-the-loop review (`review_evidence`) is required before any proposal counts toward a score, verified live. Residual risk: a future generative extraction layer (deferred, not built) would reintroduce this risk and require the citation-verification step this sprint did not need to build |
| R-2 | Sensitive OT/security evidence exposure via an unintended network call | Privacy/security | Low | Critical | Actively mitigated | No network client exists anywhere in the ingestion or assessment code path as of Sprint 2 (verified by code inspection); any future API fallback must be explicit opt-in (NFR-2) |
| R-3 | Framework version drift (C2M2/NIST updated after this project encodes them) | Product/maintenance | Medium | Medium | Mitigated by design, tested for both frameworks | Frameworks stored as versioned data (ADR-0002); both `c2m2_v2_1.yaml` and `nist_csf_2_0.yaml` cite exact source version, publisher, date, and URL (ADR-0009, ADR-0010), so a future version bump is a detectable, explicit data change. | R-4 | Scope creep across seven target frameworks | Project management | Low | Medium | Actively managed | MVP hard-scoped to two frameworks (D-1), both now delivered (C2M2 partial, NIST CSF 2.0 full); each sprint's deliverables doc confirms no scope creep occurred |

| R-10 | Retrieval quality of the interim hashed-vector embedding backend is materially worse than a neural embedder for semantic matching | AI quality | N/A — resolved | N/A | **Mitigated (closed before Sprint 3)** | Replaced with a local ONNX semantic embedder (ADR-0008). Verified with a live retrieval test: a paraphrased query sharing zero literal words with the target section ("two factor login for people connecting from outside the office") correctly ranked the "Multi-Factor Authentication" policy section first among 8 candidate chunks — the hashing backend could not have done this. Not just unit-tested; demonstrated against the running server and the real persisted vector store |
| R-11 | OneDrive-synced filesystem (`/mnt/c`) has non-instant directory-listing consistency, which can cause check-then-act race conditions | Technical (found in Sprint 1) | Medium | Medium if unaddressed elsewhere in the codebase | Partially audited; a second, concrete instance confirmed live in Sprint 10 | The recommended audit pass happened informally while writing the two new Sprint 3 data-access modules (`repositories/assessment_repository.py` predates Sprint 3; `services/framework_loader.py` is new): `framework_loader.py` does a `path.exists()` check before opening a file, structurally the same shape as the original bug, but low risk here because `framework_mapping/*.yaml` files are static and committed, not created dynamically at runtime by a concurrent process — noted as reviewed, not as a full formal audit of the whole codebase. **Sprint 10:** this same filesystem class caused a second, different symptom — Vite's default (inotify-based) dev-server file watcher silently served a stale transform of an edited component with no error, confirmed directly by inspecting the dev server's served module content before and after the fix. Mitigated for the frontend via `vite.config.ts`'s `server.watch.usePolling`, documented inline and in ADR-0016; the underlying filesystem characteristic itself remains unresolved and worth remembering for any future tooling added on this mount |
| R-12 | Nothing outside `AssessmentService` currently prevents a hypothetical direct-repository call from bypassing the finalized-assessment evidence lock | AI/data governance (found in Sprint 2 risk review) | Low (no code path currently does this) | Medium (would silently break the audit-immutability guarantee if it ever happened) | Open, unmitigated | Still open as of Sprint 5 — not addressed in Sprints 3, 4, or 5 either; carried forward honestly rather than dropped from the register; the new `review_evidence` path in Sprint 5 was checked and correctly goes through `AssessmentService`, so this sprint did not widen the gap, but it also did not close it |
| R-13 | Test suite runtime increased materially (~80s to ~95-150s) after adopting the ONNX embedding backend, because the embedder singleton is rebuilt (reloading the model) once per test client fixture rather than once per session | Technical/operational (found while adopting ADR-0008) | N/A — resolved | Low (correctness unaffected; developer feedback loop slower) | **Fixed, verified, Sprint 9** | Measured directly (constructing `LocalSemanticEmbedder()` fresh: 0.39-0.45s across three trials) before fixing, confirming the diagnosis was still correct two sprints later. Removed `get_cached_embedder` from both integration test files' per-test cache-clear list — its config never varies between tests, only `vector_store_dir`/`assessments_db_path` do, so reusing one instance across the whole session is both safe and correct. Full suite: 127.36s → 68.59s for the same 153 tests, re-verified twice. See ADR-0015 |
| R-14 | Only 2 of 10 C2M2 domains (71 of 356 practices) are transcribed as real, usable data; an assessment can only be meaningfully scored for `ASSET` and `ACCESS` | Product (found in Sprint 3) | N/A — resolved | N/A | **Closed, Sprint 10** | Deliberate, documented scope decision (ADR-0009), not an oversight; `Domain.practices_populated` makes the gap machine-readable so the API reports an honest 0 rather than a fabricated score for unpopulated domains. **Sprint 5 note:** the mapping engine correctly skips unpopulated domains when proposing candidates (verified by `test_find_mapping_candidates_skips_unpopulated_domains`), so this gap did not silently produce fabricated proposals for the 8 untranscribed domains in the meantime. **Sprint 10:** the remaining 8 domains were transcribed the same way, verified against the source document's own stated total (356 of 356 practices) — see ADR-0018. Closes backlog item US-3.1a |
| R-15 | A raw score number from `GET /assessments/{id}/score` means different things depending on the assessment's framework (a whole-number MIL 0-3 for C2M2 vs. a 0.0-1.0 coverage fraction for NIST CSF 2.0); presenting either without that context risks misinterpretation | Product/AI governance (found in Sprint 4, ADR-0010) | Low (downgraded, Sprint 6) | Medium — a coverage fraction misread as a MIL level (or vice versa) would misstate an organization's actual maturity | **Mitigated for the dashboard surface, Sprint 6** | Documented explicitly in ADR-0010 as a requirement for `executive-reporting` work; ADR-0012 directly applies it to the first real reporting surface — `OverallSummary` never averages `cumulative_mil` domain scores into a fabricated single number, only ever a count ("N of M domains at MIL1+") or, for `coverage` frameworks, a legitimate weighted fraction. No UI exists yet to enforce this visually (still text/JSON only), so the risk is not fully closed — but the one surface most likely to misuse the number (a dashboard headline) now cannot |
| R-16 (new) | Retrieval-only mapping has a real, disclosed precision ceiling: domain-general cybersecurity vocabulary produces nontrivial similarity between genuinely unrelated practice/evidence pairs near the confidence threshold | AI quality (found in Sprint 5, ADR-0011) | High (observed live, not hypothetical) | Medium — mitigated by mandatory human review before any score impact, but still creates review burden | Open, mitigated by human-in-the-loop, not by model accuracy | Directly observed in the Sprint 5 live demo: `ASSET-1a` (an asset-inventory practice) proposed at confidence 0.71 against access-provisioning policy text — a real false positive from vocabulary overlap, not a hypothetical. Threshold (0.55) was empirically calibrated (correct pairs 0.65-0.78, incorrect pairs 0.43-0.53) but the gap is real, not large. `review_evidence` requires an explicit human decision before any proposal counts toward a score, which is why this is "mitigated," not "unmitigated" — a future generative verification layer (deferred) would reduce reviewer burden, not just catch a hard failure |
| R-17 (new) | `AssessmentService` now eagerly depends on an `Embedder` for every assessment-router request (not only mapping-related ones), via `api/dependencies.py`'s `get_assessment_service()` | Technical/architectural (found in Sprint 5, ADR-0011) | Low (the embedder is `@lru_cache`'d — one-time load cost, not per-request) | Low (correctness unaffected; a transient coupling worth naming) | Open, low priority | `GET /assessments` and other non-mapping endpoints now transitively require the embedding backend to be resolvable even though they don't use it; not a functional bug today, but worth revisiting if a future sprint wants those endpoints usable without the ONNX model loaded (e.g., a lighter health-check-only mode) |

| R-18 (new, closed) | Query ordering relying implicitly on Python-side `datetime.now(UTC)` timestamps is not reliable when writes land close enough together to tie at the host clock's resolution | Technical (found and fixed in Sprint 5) | N/A — resolved | Medium (an audit-trail history returned out of order would undermine the immutability guarantee NFR-5 depends on) | **Fixed, verified stable** | Found as an intermittent, non-reproducible-on-demand test failure (`status_history` returning entries out of insertion order) while writing Sprint 5's end-to-end integration test — not found by inspection. Root-caused to timestamp-tie ordering, fixed by ordering `status_history` and `evidence_for_assessment` by SQLite's implicit `rowid` instead (ADR-0011). Verified via a direct repro script and three consecutive full-suite runs with no recurrence. SQLite-specific; flagged as needing an explicit sequence column if ADR-0007's possible future PostgreSQL migration happens |
| R-19 (new) | The dashboard's resolution list prioritizes strictly by gap count (fewest missing practices first); this is an honest effort proxy, not a business-impact score, and could be misread as "the platform recommends fixing this first for risk reasons" | Product/AI governance (found in Sprint 6, ADR-0012) | Medium (the framing is disclosed in the API response's `rationale` field, but nothing stops a downstream reader from over-interpreting it) | Low-Medium — a wrong remediation-priority read could misallocate limited compliance-team effort | Open, disclosed | Deliberate scope decision (ADR-0012): the platform has no real basis (cost data, criticality tagging) to compute an actual business-impact score, and fabricating one would violate this project's standing "verified over fabricated" rule. Each `ResolutionItem.rationale` states the effort-based reasoning in plain language rather than presenting a bare rank; revisit if `framework_mapping/` ever gains real criticality metadata |
| R-20 (new) | `AssessmentService.build_dashboard` re-fetches and re-computes the full gap analysis on every call, with no caching, same as `compute_scores` | Technical/performance (found in Sprint 6) | Low (current data volumes — max 356 C2M2 practices — make this fast; no measured latency problem) | Low | Open, unmitigated, low priority | Not optimized preemptively — no evidence yet that it needs to be, consistent with this project's practice of fixing measured problems (R-13's embedder reload) rather than hypothetical ones. Worth revisiting only if a future frontend polls this endpoint frequently |
| R-21 (new) | A downloaded PDF/XLSX report is a point-in-time snapshot; nothing prevents a board or reviewer from acting on a stale export after evidence has since been reviewed, added, or rejected | Product/AI governance (found in Sprint 7, ADR-0013) | Medium (exports are explicitly not persisted or versioned server-side — see ADR-0013 — so the platform has no record of which snapshot is in circulation) | Low-Medium — a decision made from a stale export could be based on an already-superseded gap analysis | Open, disclosed | Direct consequence of ADR-0013's no-persistence decision, made deliberately rather than accidentally: the alternative (server-side archival/versioning of every export) is new state and invalidation complexity with no current stakeholder request behind it. Both formats print a `Generated: <timestamp>` value (a header line in the PDF, a `Situation` sheet row in the XLSX) as a self-contained staleness signal that survives the file leaving the API — revisit if this platform ever supports point-in-time/recurring compliance reporting (see roadmap) |
| R-22 (new) | Chat (Sprint 8) cannot answer from a manually-linked evidence item that has no `chunk_id` (a coarse, document-level link) — only chunk-scoped, AI-proposed-then-reviewed evidence is searchable | Product (found in Sprint 8, ADR-0014) | Medium (manual links without a chunk_id are common — every manual-link demo in this project's own test suite has used one) | Low — the gap is a missed answer, not a wrong one; the evidence itself is still visible via `GET /assessments/{id}/evidence` | Open, disclosed | Deliberate scope decision (ADR-0014): a link with no chunk_id has no specific source text to compare a question against or quote back, and fabricating one would violate the citation-verification principle every other AI-facing feature in this project follows. Revisit if manual evidence linking ever gains an optional chunk-selection UI |
| R-23 (new) | Chat's similarity threshold cannot cleanly separate genuinely relevant questions from genuinely unrelated ones — verified live: the weakest true match (0.538) and the strongest observed false positive (0.542) are close enough that no threshold value would exclude one without the other | AI quality (found in Sprint 8, ADR-0014) | High (observed live, not hypothetical — same domain-general-vocabulary-overlap mechanism as R-16) | Low-Medium — mitigated by the result always being a real, cited evidence chunk (never fabricated text) and by `similarity` always being shown, not hidden behind a pass/fail cutoff | Open, disclosed, mitigated by transparency not by threshold precision | Same underlying phenomenon R-16 already documented for the mapping engine, now confirmed present in a second retrieval task (question-to-evidence, not just practice-to-evidence). The threshold (0.4) filters the clearest noise but is not claimed to eliminate borderline false positives; the reviewer, seeing the similarity score attached to every result, is the actual final judge of relevance — consistent with this project's human-in-the-loop discipline everywhere else |
| R-24 | The Docker Compose deployment stack (`deployment/`, ADR-0017) had not been run end-to-end via `docker compose up` — the authoring environment initially had no Docker installed and no passwordless `sudo` to install it | Technical/operational (found in Sprint 10) | N/A — resolved | N/A | **Closed, Sprint 10** | Docker Desktop was installed and its WSL2 integration enabled for the authoring distro, closing the environment gap. Live run found and fixed one real bug the static checks could not have caught (`npm ci` needed `--legacy-peer-deps` for the same `openapi-typescript`/TypeScript 6 peer conflict ADR-0016 already resolved for local installs). Full verification then passed: `docker compose build`/`up` succeed, `ollama` correctly stays off by default and starts correctly via `--profile ollama`, a full persona walkthrough against the real containers produced zero console errors (and correctly proposed 349 AI mapping candidates, reflecting ADR-0018's full C2M2 transcription), and data/model-cache persistence across `docker compose down`/`up` was confirmed directly. See ADR-0017 |
| R-25 (new) | Cross-framework equivalence (`framework_mapping/cross_framework_equivalence.yaml`, ADR-0019) covers 79 of 106 NIST CSF 2.0 subcategories, not all of them; and `services/framework_loader.py`'s merge logic hardcodes exactly two frameworks by column name (`c2m2_practice_id`/`nist_subcategory_id`), not a generic N-framework schema | Product/technical (found in Sprint 10, ADR-0019) | Low — both limitations are structural/by-design, not accidental | Low — the 27 excluded subcategories are each disclosed with a specific reason (genuine standards gap or C2M2's repeated Management-Activities boilerplate creating an ambiguous many-domains-match, not a review shortcut); a third framework's equivalence data simply isn't representable yet, a real but bounded gap | Open, disclosed, low priority | Deliberate scope decision, not an oversight: exhaustive coverage was rejected in favor of genuine, rationale-backed matches only (ADR-0019's own alternatives-considered section), and generalizing the schema for a third framework was rejected as building ahead of actual need (no NERC CIP/ISO 27001 equivalence work exists yet to design against). Revisit the loader's hardcoding if/when a third framework's equivalence data is actually needed |

## Review cadence

This register is updated at the end of every sprint as part of that sprint's consulting deliverables doc (see `docs/consulting/sprint-0X-deliverables.md`, Risk Assessment section) and reconciled back into this master file when a risk's status changes.
