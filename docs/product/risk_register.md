# Risk Register

Master, living register. Consolidates `PROJECT_CHARTER.md` Section 7 (initial risks) with risks found during Sprint 1-2 execution (`docs/consulting/sprint-0*-deliverables.md`). Likelihood/impact are qualitative (Low/Medium/High) — no incident data exists yet to quantify them, and presenting false-precision numbers would contradict this project's own stated documentation standard (see `PROJECT_CHARTER.md` Section 2's business problem framing).

| ID | Risk | Category | Likelihood | Impact | Status | Mitigation |
|---|---|---|---|---|---|---|
| R-1 | AI hallucinates a compliance claim not supported by evidence | AI governance | Low (downgraded, Sprint 5) | High | **Mitigated by construction, Sprint 5** | Sprint 5's mapping engine is retrieval-only, not generative (ADR-0011): a proposed mapping's "citation" is the literal retrieved chunk, never a model-generated quote, so there is nothing to hallucinate; human-in-the-loop review (`review_evidence`) is required before any proposal counts toward a score, verified live. Residual risk: a future generative extraction layer (deferred, not built) would reintroduce this risk and require the citation-verification step this sprint did not need to build |
| R-2 | Sensitive OT/security evidence exposure via an unintended network call | Privacy/security | Low | Critical | Actively mitigated | No network client exists anywhere in the ingestion or assessment code path as of Sprint 2 (verified by code inspection); any future API fallback must be explicit opt-in (NFR-2) |
| R-3 | Framework version drift (C2M2/NIST updated after this project encodes them) | Product/maintenance | Medium | Medium | Mitigated by design, tested for both frameworks | Frameworks stored as versioned data (ADR-0002); both `c2m2_v2_1.yaml` and `nist_csf_2_0.yaml` cite exact source version, publisher, date, and URL (ADR-0009, ADR-0010), so a future version bump is a detectable, explicit data change. C2M2 coverage is still partial (2 of 10 domains); NIST CSF 2.0 is fully covered (all 106 subcategories) |
| R-4 | Scope creep across seven target frameworks | Project management | Low | Medium | Actively managed | MVP hard-scoped to two frameworks (D-1), both now delivered (C2M2 partial, NIST CSF 2.0 full); each sprint's deliverables doc confirms no scope creep occurred |
| R-5 | No real enterprise data available for realistic testing | Data/credibility | N/A (accepted constraint) | Medium | Accepted, mitigated | Synthetic sample evidence documents, explicitly labeled (`data/sample_evidence/`) |
| R-6 | Local hardware/inference limitations | Technical | Low (downgraded from Sprint 0) | Medium | Mitigated, re-verified | ADR-0008's semantic embedder (ONNX, no PyTorch/CUDA) was checked against this risk before adoption, not just assumed safe — install and first-run latency measured directly (~7s cold, includes a 67MB one-time download) rather than assumed acceptable |
| R-7 | Solo-builder time constraint against a 10-sprint plan | Project management | Medium | Medium | Actively managed | Each sprint independently demoable (verified live each sprint, not just tested); PM docs were deliberately deferred rather than rushed to avoid quality loss under time pressure |
| R-8 | Non-English documents are untested and likely mishandled | Product (new, found in PM review) | Medium | Low (MVP has no non-English user identified) | Open, unmitigated | Not addressed in code; flagged here rather than silently assumed away — see Assumptions Log A-2 |
| R-9 | Development environment is not yet reproducible without manual, human-gated setup | Technical/operational | High (already occurred once) | Low | Open, partially mitigated | Sprint 1's setup steps documented in `docs/consulting/sprint-01-deliverables.md`; no automated environment bootstrap script exists yet |
| R-10 | Retrieval quality of the interim hashed-vector embedding backend is materially worse than a neural embedder for semantic matching | AI quality | N/A — resolved | N/A | **Mitigated (closed before Sprint 3)** | Replaced with a local ONNX semantic embedder (ADR-0008). Verified with a live retrieval test: a paraphrased query sharing zero literal words with the target section ("two factor login for people connecting from outside the office") correctly ranked the "Multi-Factor Authentication" policy section first among 8 candidate chunks — the hashing backend could not have done this. Not just unit-tested; demonstrated against the running server and the real persisted vector store |
| R-11 | OneDrive-synced filesystem (`/mnt/c`) has non-instant directory-listing consistency, which can cause check-then-act race conditions | Technical (found in Sprint 1) | Medium | Medium if unaddressed elsewhere in the codebase | Partially audited | The recommended audit pass happened informally while writing the two new Sprint 3 data-access modules (`repositories/assessment_repository.py` predates Sprint 3; `services/framework_loader.py` is new): `framework_loader.py` does a `path.exists()` check before opening a file, structurally the same shape as the original bug, but low risk here because `framework_mapping/*.yaml` files are static and committed, not created dynamically at runtime by a concurrent process — noted as reviewed, not as a full formal audit of the whole codebase |
| R-12 | Nothing outside `AssessmentService` currently prevents a hypothetical direct-repository call from bypassing the finalized-assessment evidence lock | AI/data governance (found in Sprint 2 risk review) | Low (no code path currently does this) | Medium (would silently break the audit-immutability guarantee if it ever happened) | Open, unmitigated | Still open as of Sprint 5 — not addressed in Sprints 3, 4, or 5 either; carried forward honestly rather than dropped from the register; the new `review_evidence` path in Sprint 5 was checked and correctly goes through `AssessmentService`, so this sprint did not widen the gap, but it also did not close it |
| R-13 | Test suite runtime increased materially (~80s to ~95-150s) after adopting the ONNX embedding backend, because the embedder singleton is rebuilt (reloading the model) once per test client fixture rather than once per session | Technical/operational (found while adopting ADR-0008) | High (already observed) | Low (correctness unaffected; developer feedback loop slower) | Open, unmitigated | Not fixed now — flagged rather than silently accepted; Sprint 5's full suite ran in ~41s (118 tests), suggesting this has not worsened further, but the underlying per-fixture reload is unchanged; a session-scoped test fixture for the embedder would resolve it without any production code change |
| R-14 | Only 2 of 10 C2M2 domains (71 of 356 practices) are transcribed as real, usable data; an assessment can only be meaningfully scored for `ASSET` and `ACCESS` | Product (found in Sprint 3) | High (known, not hypothetical) | Medium — blocks a complete C2M2 assessment, but the 2 populated domains are fully real and correctly scored | Open, tracked as backlog item US-3.1a | Deliberate, documented scope decision (ADR-0009), not an oversight; `Domain.practices_populated` makes the gap machine-readable so the API reports an honest 0 rather than a fabricated score for unpopulated domains. **Sprint 5 note:** the mapping engine correctly skips unpopulated domains when proposing candidates (verified by `test_find_mapping_candidates_skips_unpopulated_domains`), so this gap does not silently produce fabricated proposals for the 8 untranscribed domains — it compounds R-14's existing impact (a smaller proposal surface) rather than introducing a new failure mode |
| R-15 | A raw score number from `GET /assessments/{id}/score` means different things depending on the assessment's framework (a whole-number MIL 0-3 for C2M2 vs. a 0.0-1.0 coverage fraction for NIST CSF 2.0); presenting either without that context risks misinterpretation | Product/AI governance (found in Sprint 4, ADR-0010) | Medium (real once any report or dashboard surfaces these numbers, Sprint 6-7) | Medium — a coverage fraction misread as a MIL level (or vice versa) would misstate an organization's actual maturity | Open, tracked | Documented explicitly in ADR-0010 as a requirement for `executive-reporting` work; the API docstring for `/score` states the distinction; no UI exists yet to enforce it visually — recommend before Sprint 6-7 report generation begins |
| R-16 (new) | Retrieval-only mapping has a real, disclosed precision ceiling: domain-general cybersecurity vocabulary produces nontrivial similarity between genuinely unrelated practice/evidence pairs near the confidence threshold | AI quality (found in Sprint 5, ADR-0011) | High (observed live, not hypothetical) | Medium — mitigated by mandatory human review before any score impact, but still creates review burden | Open, mitigated by human-in-the-loop, not by model accuracy | Directly observed in the Sprint 5 live demo: `ASSET-1a` (an asset-inventory practice) proposed at confidence 0.71 against access-provisioning policy text — a real false positive from vocabulary overlap, not a hypothetical. Threshold (0.55) was empirically calibrated (correct pairs 0.65-0.78, incorrect pairs 0.43-0.53) but the gap is real, not large. `review_evidence` requires an explicit human decision before any proposal counts toward a score, which is why this is "mitigated," not "unmitigated" — a future generative verification layer (deferred) would reduce reviewer burden, not just catch a hard failure |
| R-17 (new) | `AssessmentService` now eagerly depends on an `Embedder` for every assessment-router request (not only mapping-related ones), via `api/dependencies.py`'s `get_assessment_service()` | Technical/architectural (found in Sprint 5, ADR-0011) | Low (the embedder is `@lru_cache`'d — one-time load cost, not per-request) | Low (correctness unaffected; a transient coupling worth naming) | Open, low priority | `GET /assessments` and other non-mapping endpoints now transitively require the embedding backend to be resolvable even though they don't use it; not a functional bug today, but worth revisiting if a future sprint wants those endpoints usable without the ONNX model loaded (e.g., a lighter health-check-only mode) |

| R-18 (new, closed) | Query ordering relying implicitly on Python-side `datetime.now(UTC)` timestamps is not reliable when writes land close enough together to tie at the host clock's resolution | Technical (found and fixed in Sprint 5) | N/A — resolved | Medium (an audit-trail history returned out of order would undermine the immutability guarantee NFR-5 depends on) | **Fixed, verified stable** | Found as an intermittent, non-reproducible-on-demand test failure (`status_history` returning entries out of insertion order) while writing Sprint 5's end-to-end integration test — not found by inspection. Root-caused to timestamp-tie ordering, fixed by ordering `status_history` and `evidence_for_assessment` by SQLite's implicit `rowid` instead (ADR-0011). Verified via a direct repro script and three consecutive full-suite runs with no recurrence. SQLite-specific; flagged as needing an explicit sequence column if ADR-0007's possible future PostgreSQL migration happens |

## Review cadence

This register is updated at the end of every sprint as part of that sprint's consulting deliverables doc (see `docs/consulting/sprint-0X-deliverables.md`, Risk Assessment section) and reconciled back into this master file when a risk's status changes.
