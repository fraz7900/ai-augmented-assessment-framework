# ADR-0012: Executive dashboard is templated, not generated, and never averages an ordinal MIL score

**Status:** Accepted
**Sprint:** 6
**Deciders:** Fraz Ahmed
**Related:** `.claude/skills/executive-reporting/SKILL.md`, `services/report_service.py`, `services/scoring_service.py`, `docs/product/risk_register.md` (R-15)

## Context

Sprint 6 is the first half of the two-part Executive Reporting epic (`executive-reporting` skill's "Example usage" note: Sprint 6 is dashboard/analytics, Sprint 7 is PDF/XLSX generation). The skill mandates a situation/complication/resolution structure, MECE grouping of gaps, a "so what" sentence attached to every number, and a visible distinction between AI-proposed and human-verified content. None of this requires an LLM to produce — every input (domain scores, evidence counts, which practices are missing) is already computed, structured data the platform holds. Two design questions specific to this data, not addressed by the skill itself, had to be resolved before writing `report_service.py`:　how to summarize progress across an entire framework in one headline, and whether to generate any of the narrative text with a model now that a "report" is finally being produced.

## Decision

1. **The dashboard is entirely templated from real computed values — no LLM, no generated prose.** `_domain_so_what` and the resolution `rationale` strings are Python f-strings built from actual counts (met/total practices, current MIL, coverage fraction), not a model's output. This sprint still has no LLM in the loop at all (retrieval-only mapping engine from ADR-0011 is unchanged); the dashboard does not change that.
2. **The overall-progress headline never averages a cumulative_mil framework's per-domain scores.** For `cumulative_mil` frameworks (C2M2), `OverallSummary.headline` reports a count — "N of M assessable domains have reached MIL1 or higher" — never a mean of the domain MIL values. For `coverage` frameworks (NIST CSF 2.0), where each domain's score is already a true 0.0-1.0 fraction of a count, a practice-weighted average across domains is computed and reported as `overall_coverage_fraction`.
3. **Unpopulated domains (C2M2's still-untranscribed 8) are excluded from both the complication list and the overall headline's denominator, but always named in `situation.unpopulated_domains`.** Silently omitting them would make an incomplete assessment look complete; silently including them (as if MIL0 reflected a real gap analysis) would fabricate a gap list against practices that don't exist in the schema yet. Naming them explicitly, in both places, was the only option that is honest in both directions.

## Rationale

1. **MIL is an ordinal scale, not an interval one — the c2m2-expert skill's own semantics forbid averaging it.** A domain at MIL1 is not "half as mature" as one at MIL2; the levels are cumulative gates, not equally-spaced points. Averaging `[1, 0, 2, 0]` into `0.75` and presenting it as "average maturity" would misrepresent the underlying scale exactly the way R-15 in the risk register warns against — R-15 was about domain-level scores being ambiguous across frameworks; this decision extends the same discipline to an even more visible surface, the assessment's single headline number.
2. **A weighted average is legitimate for coverage frameworks because the underlying quantity actually is a fraction of a count.** `compute_domain_coverage` already returns `covered / total` per domain; summing covered and total across domains before dividing is the same operation at a larger grain, not a new kind of number. This is why the two scoring models get genuinely different summary logic rather than one function trying to handle both — the same "dispatch on declared model, not name" principle `scoring_service.py` already uses.
3. **No generated narrative, because nothing about this sprint's scope required one, and the project has a standing rule against fabricating AI capability.** The `executive-reporting` skill's structuring principle is a content-organization rule, not a mandate that the content be model-written. Every sentence the dashboard emits is directly traceable to a real number computed two lines earlier in `report_service.py` — there is nothing for a citation-verification step to check, because nothing is generated. This mirrors ADR-0011's citation reasoning: a design that has no generated claim structurally cannot hallucinate one.
4. **"Quick wins" (fewest missing practices first) is the resolution list's prioritization, not a fabricated impact score.** The project has no real basis yet for a business-impact weighting (no cost data, no criticality tagging in `framework_mapping/`) — inventing one would be exactly the kind of unverifiable number this project's standing discipline refuses to present. Effort (gap count) is the one dimension the platform can honestly quantify today; `_build_resolution`'s rationale field says exactly that, not more.

## Consequences

- `models/report.py` is new: a set of read-only Pydantic response shapes (`DashboardReport`, `Situation`, `DomainGapGroup`, `GapItem`, `ResolutionItem`, `OverallSummary`), never persisted, always computed fresh per request — consistent with how `models/framework.py`'s `FrameworkDefinition` is read-only reference data, not a database table.
- `GET /assessments/{id}/dashboard` reuses the exact same `AssessmentNotFoundError` / `FrameworkScoringUnavailableError` exception-to-HTTP mapping as `GET /assessments/{id}/score`, since it depends on the same two inputs (framework schema, evidence links) and can fail the same two ways.
- No PDF/XLSX export exists yet — Sprint 7's explicit scope, per the `executive-reporting` skill's stated split. The dashboard is API-only, consistent with this project's backend-first sequencing every sprint so far (no frontend exists).
- The `has_pending_ai_proposal` flag on each `GapItem` is the first place NFR-4's data-model-level AI-proposed/human-verified distinction is surfaced through an aggregate view, not just a per-link field — closing part of NFR-4's previously-noted "UI-level requirement pending frontend" gap at the API-response level, though an actual UI still does not exist.

## Alternatives considered

- **Average MIL scores into a single 0-3 "overall maturity" number.** Rejected — the ordinal-scale argument above; this is the single most likely number a non-technical reader would misinterpret if presented without the caveat, which is exactly the failure mode R-15 was written to flag before it could happen.
- **Generate the "so what" sentences with the ONNX embedder repurposed as a text generator, or by adding a lightweight template-selection model.** Rejected — the embedder is a similarity model, not a generator, and using it that way would be a dishonest capability claim; a template-selection model would be complexity with no benefit over the deterministic f-strings actually used, since the inputs (met/total, MIL, coverage fraction) are already a small, enumerable set of cases.
- **Give unpopulated C2M2 domains a placeholder complication entry ("no data").** Rejected in favor of the current design: listing them once, honestly, in `situation.unpopulated_domains` communicates the same fact without cluttering the complication section (which is meant to be actionable gaps within scope) with entries a reviewer cannot act on yet.
