# ADR-0011: Sprint 5's mapping engine is retrieval-only, not generative — Ollama deferred

**Status:** Accepted, explicitly interim
**Sprint:** 5
**Deciders:** Fraz Ahmed
**Related:** `.claude/skills/evidence-extraction/SKILL.md`, `.claude/skills/assessment-generation/SKILL.md`, `.claude/skills/framework-mapping/SKILL.md`, `services/mapping_service.py`, `docs/architecture/01-claude-code-workspace.md` (MCP/Ollama section)

## Context

`PROJECT_CHARTER.md` and the Sprint 0 architecture docs describe "local LLM reasoning" as a core capability, and `ai/README.md` has planned `ai/model_client.py` (an Ollama/API client abstraction) since Sprint 0. No sprint through Sprint 4 has actually stood up Ollama — embeddings (ADR-0006, ADR-0008) covered retrieval, not generation. Sprint 5 ("Framework mapping engine, AI-proposed evidence linking") is the first sprint whose charter description implies generative reasoning: matching evidence to practices, in the style the `evidence-extraction` skill describes — retrieve, ask a model to extract a claim and quote its source, verify the quote actually appears in the source.

Before deciding scope, actual feasibility of installing Ollama in this development environment was checked, not assumed: the standard install script (`ollama.com/install.sh`) requires `sudo` throughout (creating a system user, a systemd service, writing to `/usr/local/bin`) — this environment has already established (Sprint 1's venv setup) that `sudo` is not available without the project sponsor's direct, out-of-band intervention. A sudo-free path exists (GitHub release tarball extracted to a user-owned directory), but the current release's Linux package is 1.4 GB — over an order of magnitude larger than any dependency this project has installed so far (the previous largest, `onnxruntime`, was under 20 MB) — before even pulling a model (typically several hundred MB to a few GB more).

## Decision

Sprint 5 ships a fully real, working **retrieval-only** mapping engine: for each not-yet-covered practice in an assessment's framework, embed the practice's text (reusing the existing `Embedder` from ADR-0008), search the assessment's already-associated documents in the vector store, and propose an `EvidenceLink` when the best match's similarity exceeds an empirically-set threshold. No LLM generates a claim or a quote. Ollama-based generative extraction is deferred, not abandoned — see Consequences.

## Rationale

1. **The evidence-extraction skill's own confidence rule already points this direction.** It requires confidence to come from "retrieval similarity score and verification pass/fail... not from the model's own self-reported confidence." A pure-retrieval engine's confidence signal (cosine similarity, `distance_to_confidence` in `services/mapping_service.py`) is exactly that class of signal — this sprint implements the confidence half of that design directly, without yet needing the generation half.
2. **No possibly-hallucinated quote means the citation-verification gate is trivially satisfied, not skipped.** The skill's non-negotiable rule — every extracted claim must carry a citation verified to actually appear in the source — is automatically true here because the "evidence" proposed IS the literal retrieved chunk, not a model's paraphrase or quotation of it. This is stated explicitly, not glossed over: a future generative layer reintroduces the exact risk this sprint currently cannot have, and will need the verification step the skill describes, which this sprint does not yet build.
3. **Cost-checked, not cost-assumed, before deciding against Ollama this sprint.** The 1.4 GB package size and the specific `sudo` requirement were confirmed by actually fetching the install script and inspecting it, and by querying the current GitHub release's asset sizes — the same "verify before deciding" discipline ADR-0009/ADR-0010 applied to framework source documents, applied here to an infrastructure decision instead.
4. **Document-scoped retrieval, not global-corpus retrieval, is a real design decision, not a simplification.** `services/mapping_service.find_mapping_candidates` only searches documents already connected to the assessment via at least one existing evidence link (`VectorRepository.search_within_documents`, new this sprint). Searching the entire vector store would let one assessment's proposals cite evidence uploaded for a completely unrelated assessment — a real cross-assessment data-leakage-shaped bug this design avoids structurally, not by convention.
5. **Threshold calibrated empirically against real text, not chosen arbitrarily.** Before picking `mapping_similarity_threshold = 0.55`, real C2M2 practice text was embedded against real synthetic policy chunk text and the resulting cosine similarities inspected directly: correct pairings scored 0.65-0.78, clearly incorrect pairings (an incident-recovery practice against access-control policy text) scored 0.43-0.53. The gap is real but not large — domain-general cybersecurity vocabulary produces a nontrivial similarity floor between almost any two policy-ish sentences, which is disclosed as a known precision limitation below, not hidden.

## The human-in-the-loop half: evidence review

Proposing evidence without a way to review it would violate the `assessment-generation` skill's human-in-the-loop invariant by construction. This sprint also adds `AssessmentService.review_evidence` (`POST /assessments/{id}/evidence/{evidence_id}/review`): a `PENDING` link can be accepted, rejected, or edited (correcting the AI-proposed `practice_reference` to the human's actual judgment) exactly once — reviewing an already-reviewed link is rejected (`EvidenceAlreadyReviewedError`), and review is blocked on a finalized assessment, mirroring `link_evidence`'s existing immutability rule.

## A real bug found and fixed while building this

Writing the end-to-end integration test for propose → review → score surfaced an intermittent, non-reproducible-on-demand failure: `status_history` occasionally returned status-change entries out of insertion order. Root cause: `AssessmentStatusChange.changed_at` and `EvidenceLink`'s prior lack of any `ORDER BY` both relied on Python-side `datetime.now(UTC)` timestamps (set at object construction) as an implicit ordering key — not reliable when two writes land close enough together to tie at whatever resolution the host clock actually provides. Fixed by ordering both `status_history` and `evidence_for_assessment` by SQLite's implicit `rowid` instead, which is monotonically increasing per insert regardless of wall-clock behavior — verified with a direct repro script before and after, and confirmed stable across three consecutive full test-suite runs afterward, not merely assumed fixed from reading the diff. This is disclosed here, not silently fixed and omitted, per the project's standing practice (see ADR-0007's LanceDB race-condition precedent from Sprint 1).

## Consequences

- `AssessmentService` now optionally depends on an `Embedder` (`embedder: Embedder | None = None`); `api/dependencies.py`'s `get_assessment_service()` now eagerly resolves `get_cached_embedder()` for every assessment-router request, not only mapping-related ones. Because that dependency is `@lru_cache`'d, the ONNX model only loads once per server process — a one-time cost on the first assessment API call in a fresh process, not a per-request cost — but it is a real, if minor, coupling worth naming: the assessment router now transitively requires the embedding backend to be available even for endpoints (like `GET /assessments`) that don't use it.
- US-5.2 (flagging evidence that satisfies practices across multiple frameworks) is explicitly deferred, not delivered this sprint — no `framework_mapping/cross_framework_equivalence.yaml` exists yet. Logged as an open backlog item, not silently dropped.
- Retrieval-only matching has a known, disclosed precision ceiling (R-16, see risk register): domain-general similarity means some false positives are expected near the threshold, and it cannot catch a semantic match phrased in genuinely dissimilar vocabulary the way a generative, reasoning-capable model might. A future Ollama-based extraction layer would sit on top of this retrieval step (better candidate selection feeding a generative verification/quote-extraction pass), not replace it — this sprint's retrieval layer remains useful either way.
- The Ollama feasibility check performed here (sudo requirement, 1.4 GB package size) is now a documented reference point for when this decision is revisited, rather than requiring the same investigation to be repeated from scratch.

## Alternatives considered

- **Install Ollama via the sudo-requiring script, asking the project sponsor to run it in a real terminal (the same pattern used for `python3-venv` in Sprint 1):** rejected for this sprint specifically because of scale, not principle — a 1.4 GB+ download and a long-lived background daemon is a materially larger ask than the prior `sudo apt install python3.12-venv`, and a fully real, useful retrieval-only engine was achievable without it.
- **Fabricate an LLM-style "confidence" or generated "rationale" string without an actual model behind it:** rejected outright — this is exactly the class of dishonest AI-capability claim this project has consistently refused to make (see ADR-0006, ADR-0009, ADR-0010's shared "verified over fabricated" standard).
- **Skip document-scoping and search the whole vector store, filtering client-side by relevance alone:** rejected — relevance-only filtering does not prevent one assessment's proposals from citing a different assessment's uploaded evidence, which is a privacy/correctness concern this project's own charter (Section 7) takes seriously.
