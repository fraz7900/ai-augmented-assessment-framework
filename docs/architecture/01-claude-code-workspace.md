# Claude Code Workspace Design

**Sprint:** 0, updated Sprint 1
**Status:** Hooks partially active (3 of 8), Skills active, MCP documented only
**Related:** `docs/adr/ADR-0003-progressive-hook-activation.md`, `docs/adr/ADR-0004-mcp-integration-sequencing.md`, `.claude/settings.json`, `.claude/skills/`

This project is being built inside Claude Code, so the `.claude/` workspace is itself a piece of the system's engineering, not incidental tooling. It is designed with the same standard applied to the rest of the repository: every hook, skill, and MCP recommendation exists because a specific risk or workflow in `PROJECT_CHARTER.md` requires it.

## Hooks

Claude Code hooks fire on real lifecycle events (`SessionStart`, `SessionEnd`, `PreToolUse`, `PostToolUse`, matched against tool name patterns) and run a shell command. The table below specifies every hook this project needs across its full lifetime; **Status** marks what is actually enabled today versus designed-and-deferred, per ADR-0003.

| # | Hook | Event / Matcher | Purpose | Inputs | Outputs | Why it improves this project | Status |
|---|---|---|---|---|---|---|---|
| 1 | Session banner | `SessionStart` | Orient a fresh Claude Code session against current sprint, charter, and the local-first constraint before any work happens | none | Printed banner: current sprint number, link to `PROJECT_CHARTER.md`, one-line reminder that evidence content must not leave the local machine | This repository is driven by a long, dense master prompt; a session that starts cold benefits from a forced, automatic context anchor rather than relying on the operator to re-paste context | **Active** |
| 2 | Secret scan on commit | `PreToolUse`, matcher: `Bash` where command matches `git commit` | Block a commit that contains an obvious credential (API key pattern, private key header, `.env` content) | The proposed git commit's staged diff | Allow or block, with a printed reason on block | Direct mitigation for the charter's own guidance to double-check contents before pushing, made automatic instead of relying on manual review every time | **Active** |
| 3 | Lint on save | `PostToolUse`, matcher: `Edit\|Write` on any `*.py` file | Run `ruff check` immediately after a Python file is written, using the venv's own ruff so behavior always matches CI/local runs | Changed file path | Lint output; does not block, surfaces immediately | Fast feedback loop while building the ingestion/mapping/scoring services; catches style and correctness issues before they compound across a sprint | **Active as of Sprint 1** — `backend/pyproject.toml` and ruff config now exist, so ADR-0003's activation condition is met; script is `.claude/hooks/lint-on-save.sh` |
| 4 | Test on save | `PostToolUse`, matcher: `Edit\|Write` on `backend/src/compliance_platform/**/*.py` | Run the `pytest` module(s) affected by the changed file | Changed file path, mapped to its test module | Pass/fail summary | Keeps the service-layer test suite (see `tests/README.md`) honest sprint over sprint instead of accumulating untested logic | Deferred — Sprint 1 for ingestion module tests, expands each sprint thereafter |
| 5 | Build check | `PreToolUse`, matcher: `Bash` where command matches `git commit`, scoped to changes touching `frontend/**` | Run `npm run build` before allowing a commit that touches frontend code, so a broken build is never committed | Changed frontend files | Build pass/fail | Prevents the classic "works in dev server, broken in build" gap from ever reaching version control | Deferred — Sprint 6, when `frontend/` has a real build target |
| 6 | Security scan (dependency) | `PostToolUse`, matcher: `Edit\|Write` on `backend/pyproject.toml` or `frontend/package.json` | Run a dependency vulnerability check (`pip-audit` / `npm audit`) when dependencies change | Changed manifest file | Vulnerability report | This platform's entire value proposition is trustworthiness for critical-infrastructure data; a dependency-introduced vulnerability undermines that narrative directly, so it is checked at the moment a dependency is added, not just at release time | Deferred — Sprint 1 (backend), Sprint 6 (frontend) |
| 7 | PII / synthetic-data validation | `PreToolUse`, matcher: `Write` targeting `data/sample_evidence/**` | Scan content being written into the tracked sample-evidence directory for patterns resembling real PII or a real, non-allowlisted organization name | File content about to be written | Allow, or block with a printed reason and a pointer to `data/sample_evidence/README.md`'s synthetic-data rule | This is the structural enforcement, at the tool-use layer, of the charter's Constraint that only public or synthetic data may ever enter this repository — the same principle `.gitignore` enforces at the commit layer (see ADR-0002), applied one layer earlier | Deferred — the scope narrowed during Sprint 1: the ingestion pipeline as actually built never writes raw uploaded content to `data/raw/` or `data/processed/` (uploads are processed in memory and only chunks/vectors land in the gitignored LanceDB store), so those two paths no longer need this guard. Only manually-authored additions to the tracked `data/sample_evidence/` still need it |
| 8 | Report-generation audit log | `PostToolUse`, matcher: `Bash` or tool call invoking the report generation service | Append an audit entry (timestamp, assessment ID, report format) to a local, gitignored audit log whenever a report is generated | Report metadata | Appended audit log line | Directly serves the Internal Audit and External Assessor stakeholders in the charter's Stakeholder Map, who need a traceable record of when and how a report was produced, independent of the report's own content | Deferred — Sprint 7 |

A `SessionEnd` hook (append a short "what changed this session" line to a local, gitignored dev log) was considered and designed but intentionally **not** activated in Sprint 0 either, to keep the active hook set to exactly the two with zero dependency on code that doesn't exist yet (see ADR-0003). It is the first candidate to activate once Sprint 1 begins producing enough change volume per session for a log to be useful.

## Skills

Skills are modular, on-demand domain knowledge and conventions loaded into a Claude Code session via `.claude/skills/<name>/SKILL.md`. They exist so that domain expertise (what C2M2's MIL levels actually mean; what counts as PII in this project) is written down once and consistently applied across every future sprint's session, rather than re-explained in prose each time or, worse, silently drifting.

| Skill | Responsibilities | Inputs | Outputs | Example usage |
|---|---|---|---|---|
| `c2m2-expert` | Ground any C2M2-related work in the model's actual domain/practice/MIL structure | A task touching `framework_mapping/c2m2_*.yaml` or C2M2 scoring logic | Correct domain names, MIL 0-3 definitions, practice numbering conventions applied consistently | Invoked automatically when building the C2M2 loader in Sprint 3 |
| `nist-csf-expert` | Same role for NIST CSF 2.0's six functions and category/subcategory structure | A task touching `framework_mapping/nist_csf_2_0.yaml` | Correct function names (Govern, Identify, Protect, Detect, Respond, Recover) and structure | Invoked when building the NIST CSF loader in Sprint 4 |
| `framework-mapping` | Engineering conventions for how a new framework is added to the schema and mapping engine, independent of which framework | A task adding or modifying framework support | A framework addition that follows ADR-0002's data-as-code pattern | Invoked in Sprint 5 (mapping engine) and any future NERC CIP/ISO 27001 addition |
| `evidence-extraction` | Conventions for how the AI layer extracts candidate evidence statements from a document and attaches source citations | A task in `backend/src/compliance_platform/ai/` touching extraction | Extraction logic that always preserves a verifiable source citation, never a bare claim | Invoked building `ai/retrieval.py` in Sprint 1-2 |
| `assessment-generation` | Conventions for assessment state, scoring, and the audit-trail requirement from the Stakeholder Map | A task in `services/scoring_service.py` or `assessments/` schema | Scoring logic that always carries evidence linkage, never an unexplained number (Section 4 of the charter) | Invoked in Sprint 2-4 |
| `prompt-engineering` | Standards for versioning, structuring, and evaluating prompts in `prompts/` | A task adding or modifying a prompt template | A new versioned prompt file plus an evaluation note, not an in-place edit (see ADR-0002) | Invoked whenever a mapping or extraction prompt changes |
| `executive-reporting` | Conventions for translating control-level detail into a leadership-ready maturity narrative, McKinsey/BCG-style structuring (situation-complication-resolution, MECE gap grouping) | A task in `services/report_service.py` or dashboard components | Reports and dashboard copy that read as an executive deliverable, not a control-ID dump | Invoked in Sprint 6-7 |
| `data-cleaning` | Conventions for metadata extraction, chunking, and validation quality in the ingestion pipeline | A task in ingestion or `data/processed/` schema | Consistent chunking/metadata rules across all document types the pipeline supports | Invoked in Sprint 1 |
| `document-parsing` | Format-specific parsing conventions (PDF, DOCX) and their known failure modes (e.g., PDF text-extraction artifacts) | A task adding a new document parser | Parsing logic with explicit handling for known failure modes rather than silent data loss | Invoked in Sprint 1 |
| `privacy-protection` | The project's non-negotiable rule set: local-first by default, synthetic/public data only, explicit opt-in for any cloud API call | Any task touching `ai/model_client.py`, `data/`, or anything that could transmit evidence content externally | A design or code change that never silently sends evidence to a cloud API | Invoked whenever the API fallback path (Section 12 of the charter) is touched, and referenced by hook #7 above |
| `energy-cybersecurity` | Domain context connecting features back to utilities, generation, transmission, OT/IT integration, and why this matters to a consulting audience | Any task where a feature needs to be explained in energy-sector terms | Feature descriptions and documentation that tie back to the sector context, consistent with the charter's positioning | Invoked when writing documentation, demo scripts, or MBA/consulting narrative sections |

All eleven skills are created now, in Sprint 0, even though most of their subject matter (the actual mapping engine, ingestion pipeline, etc.) doesn't exist until later sprints. This is different from the hooks decision in ADR-0003: a skill costs nothing to have present and unused (it is only loaded into context when relevant), while a hook that fires automatically can actively fail or produce noise before its dependencies exist. That asymmetry is why skills are front-loaded and hooks are not.

## Model Context Protocol (MCP) integrations

Per ADR-0004, this section is a design recommendation for future sprints, not active configuration. No project-specific MCP server is wired into `.claude/settings.json` as of Sprint 0.

| MCP server | Why needed | Expected workflow | Security considerations | Activation point |
|---|---|---|---|---|
| Local filesystem | Read/write access scoped to the repository for Claude Code's own file operations | Already effectively provided by Claude Code's built-in file tools; a dedicated MCP filesystem server would only matter if scope needed to extend beyond the repo (e.g., reading a sample evidence corpus stored elsewhere on disk) | Scope must stay repository-local; never point at a directory containing real (non-synthetic) organizational data | Only if/when sample data needs to be sourced from outside the repo |
| Git | Structured git operations (branch, commit, diff inspection) beyond ad hoc shell commands | Used for sprint-boundary commits and, later, PR creation once the repo has a remote | Same permission model as direct git shell access; no additional exposure beyond what's already available | Available now via existing tooling; a dedicated server adds value once the workflow includes PR-based review |
| Ollama | Local model invocation for ingestion, extraction, mapping, and scoring | The core inference path for the entire platform, called from `backend/src/compliance_platform/ai/model_client.py` | Runs entirely on localhost; no data leaves the machine, which is the load-bearing security property of the whole product | Sprint 1, as soon as the ingestion pipeline needs to call a model |
| Local embeddings | Generating vector representations of document chunks for retrieval | Called by `ai/embeddings.py`, feeding into whichever vector store is chosen | Same locality property as Ollama; embedding model choice should be validated to run fully offline | Sprint 1 |
| ChromaDB or LanceDB | Vector storage and similarity search over embedded evidence chunks | Called by `repositories/vector_repository.py` | Local file-based store in MVP; no network exposure by default, but must be explicitly configured to stay that way if either tool's server mode is ever used | Sprint 1, after the choice between the two is made per `docs/architecture/00-repository-architecture.md` |
| SQLite | Relational storage for assessment state, evidence metadata, and the audit trail, in the MVP's single-user local deployment | Called by `repositories/assessment_repository.py` and `evidence_repository.py` | File-based, local by default; the natural MVP choice given the charter's single-user assumption | Sprint 2 |
| PostgreSQL | Relational storage if/when the platform moves beyond single-user local deployment | Would replace SQLite behind the same repository interface (see `repositories/README.md`) | Requires real credential management and network access controls the MVP explicitly avoids needing | Deferred to the multi-tenant roadmap item in `PROJECT_CHARTER.md` Section 13, not the MVP |
| PDF parser | Structured text and metadata extraction from uploaded PDF evidence | Called by `ai/` or a dedicated ingestion module during document processing | Must run entirely locally like every other step in the ingestion path; parser choice should be checked for any telemetry/network calls before adoption | Sprint 1 |

## Summary of what's live today

- **Active:** `SessionStart` banner, `PreToolUse` secret-scan-on-commit, `PostToolUse` lint-on-save (3 hooks, as of Sprint 1); all 11 skills.
- **Documented, not yet active:** the remaining 5 hooks (activate per the sprint noted in the table above); all 8 MCP integrations (activate per the sprint noted in the table above — the app's own runtime use of `lancedb`/`pypdf` as Python libraries is not the same thing as a Claude Code MCP integration for those tools, and remains a separate, still-deferred decision).

This split — plan everything now, activate what's safe now, defer the rest with an explicit trigger condition — is the same discipline the charter applies to MVP scoping (Section 12) applied one level down, to tooling.
